    1.3 Concurrency Model¶

    Module Primitives Key Patterns
    memory sync.Mutex (jobsMu, indexer.mu, watcher.mu, cache.mu), sync.WaitGroup (scheduler), chan (watcher, embed grpc) One goroutine per background task (decay scheduler, file watcher, background indexer). Reader-mutex for tiered cache.
    broker-go sync.Mutex (proxy.mu, proxy.writeMu), sync.RWMutex (registry.mu), chan (proxy.pending, proxy.readerCh), sync.WaitGroup (proxy reader), select with timeout (HealthDeep ping) Single goroutine async reader per proxy. Mutex-guarded pending map for request routing. Channel-based worker pool for parallel dispatch.
    orchestrator-go sync.Mutex (Orchestrator.mu, FileOwnershipRegistry.mu, context builder), sync.RWMutex (ProtocolMachine), sync.WaitGroup (TaskRunner worker pool), chan (TaskRunner jobs/results) Worker pool pattern: N workers pull from jobs channel, push to results channel. Protocol machine uses RWMutex for state access.
    backend-go None No concurrency — thin serialization layer.
    TUI Node.js EventEmitter, Promise.allSettled (parallel dispatch) Event-driven. Compaction orchestrator uses boolean mutex lock (not a real mutex). Parallel dispatch up to 8 agents.
