    Roadmap: Neuralgentics v0.1.3 — P2 Phase (Continuity + Offline)¶

    Author: boomerang-architect (deepseek-v4-pro:cloud) Date: 2026-06-05 Status: v0.1.3 P2 (5.5 days) — DESIGN PHASE ONLY Builds on: P0 (13 cards, all done) + P1 (9 cards, all done) + Session 25/26 cleanup

    P2 delivers cross-session continuity, offline mode, and panel upgrades. When P2 ships, the TUI survives restarts: session state is checkpointed to neuralgentics, token spend persists, model preferences stick, opportunity findings are cached, and the TUI is usable with minimal functionality even when the OpenCode server or Go backend is down. The diff panel graduates from 2-way to 3-way merge.

    Correction from original P0/P1 roadmap: The P0 roadmap (v0.1.0) listed P2 as "Slash Commands + Cross-Session" (~4 days) with agent preferences, multi-session continuity, heartbeat + stale detection, event synthesis, and HANDOFF.md auto-generation. That was inaccurate — slash commands shipped in P1 (T-039, 10 functional), so P2 focuses on persistence, offline resilience, and viewer upgrades. Heartbeat/stale detection and HANDOFF.md auto-generation are deferred to P3 (hardening). Agent preferences are replaced by model-selection persistence (T-082) since the model registry (T-025) already handles task-scoped routing.

    Top-Level Summary¶

    P0 gave us a working pipeline (TUI + compaction + dispatch + verify). P1 gave us polish (themes, slash commands, token accounting, opportunity detection, kanban with circuit breaker, aggregator-aware lookup, permission gating). Both phases are fully shipped with 650 tests passing.

    The gap: nothing survives a TUI restart. The token counter resets to zero. The opportunity detector's 8-pattern findings evaporate. The model choice reverts to defaults. The compaction checkpoints (gemma4:31b's extracted facts + reseed snapshot) are lost. The diff panel can only show 2-way changes. And if the Go backend or OpenCode server goes down, the TUI is a dead terminal — no read-only fallback.

    P2 closes these gaps with 7 cards. The foundation is checkpoint persistence (T-079) — serialize compaction cycle output to neuralgentics. On top of that, session resume (T-080) restores the TUI to its previous state on restart, offline mode (T-081) provides read-only degraded operation, model-selection persistence (T-082) remembers the sticky per-session model choice, spend history (T-084) and offline opportunity cache (T-085) persist their data across sessions, and 3-way merge (T-083) upgrades the diff panel.

    Note on "/spend & /opportunities go from stub to live": Per the card spec (T-ARCH-P2), this was listed as a P2 theme. However, these panels are already live — T-033 wired TokenCounter to SpendPanel, T-034 wired OpportunityDetector to the /opportunities command, and T-039 made both functional. P2 adds persistence to both (T-084 and T-085) so their data survives restarts. The P2 cards are correctly scoped to persistence, not initial implementation.

    P2 Task Breakdown¶

    Task T-079: Checkpoint Persistence¶

    • Goal: Persist compaction cycle output (extracted facts, reseed snapshot, token state, session metadata) as type: "compaction_checkpoint" memories in neuralgentics. On TUI restart, load the most recent checkpoint to restore session state. This is the foundation card for all persistence features (T-080, T-081, T-082, T-084, T-085).
    • Scope IN:
    • packages/tui/src/compaction/writer.ts — extend CompactionWriter.writeCheckpoint() to serialize a CompactionCheckpoint struct (facts extracted, facts count, tokens before/after, timing, session ID, timestamp, confidence scores) into a JSON blob and call neuralgenticsClient.memory.add() with type: "compaction_checkpoint".
    • packages/tui/src/compaction/types.ts — add CompactionCheckpoint interface: {checkpointId: string, sessionId: string, timestamp: string, factsExtracted: number, tokensBefore: number, tokensAfter: number, savingsRatio: number, reverted: boolean, reseeded: boolean, confidenceScores: Map<string, number>, extractedMemoryIds: string[]}.
    • packages/tui/src/compaction/orchestrator.ts — extend CompactionOrchestrator.compact() to call writer.writeCheckpoint() after successful extraction + writing. Return checkpoint ID in CompactionResult.
    • packages/tui/src/session/session-manager.ts — add loadLastCheckpoint(): Promise<CompactionCheckpoint | null> method that queries neuralgentics for most recent type: "compaction_checkpoint" memory (sorted by timestamp DESC, limit 1). Add restoreFromCheckpoint(checkpoint: CompactionCheckpoint): Promise<void> that repopulates session state.
    • packages/tui/src/neuralgentics-client/client.ts — verify memory.query supports filter: {type: "compaction_checkpoint"} and sort: DESC (the Go backend's QueryMemories method supports sourceType filtering; verify or add this filter path).
    • packages/tui/src/__tests__/checkpoint-persistence.test.ts — unit tests: write checkpoint, read back, verify all fields round-trip, handle missing checkpoint gracefully, handle corrupted checkpoint (bad JSON), verify session manager loads most recent.
    • Scope OUT:
    • Restoring the FULL TUI state from checkpoint (→ T-080 wires all the panels to checkpoint restore)
    • Persisting token counter state (→ T-084 owns its own persistence logic)
    • Persisting opportunity detector state (→ T-085 owns its own persistence logic)
    • Persisting model preferences (→ T-082 owns its own persistence logic)
    • Sub-tasks:
    • packages/tui/src/compaction/types.ts — add CompactionCheckpoint interface (~30 lines)
    • packages/tui/src/compaction/writer.ts — extend with writeCheckpoint(), call neuralgenticsClient.memory.add() (~40 lines add)
    • packages/tui/src/compaction/orchestrator.ts — call writer.writeCheckpoint() after writer.writeFacts(), add checkpoint ID to CompactionResult (~20 lines add)
    • packages/tui/src/session/session-manager.ts — add loadLastCheckpoint() and restoreFromCheckpoint() (~60 lines add)
    • packages/tui/src/neuralgentics-client/client.ts — verify/ensure memory.query can filter by sourceType + sort by timestamp (investigate + ~10 lines if needed)
    • packages/tui/src/__tests__/checkpoint-persistence.test.ts — 12 tests (write, read, round-trip, missing, corrupted, most-recent, empty checkpoint, concurrent checkpoints, restore from checkpoint, invalid JSON, edge cases)
    • Acceptance criteria:
    • After compaction cycle completes, a type: "compaction_checkpoint" memory exists in neuralgentics with all fields populated correctly
    • SessionManager.loadLastCheckpoint() returns the most recently created checkpoint
    • Checkpoint includes extracted memory IDs (so T-080 can reload them on resume)
    • No checkpoint exists → loadLastCheckpoint() returns null gracefully (no crash)
    • Corrupted checkpoint JSON → logged as warning, treated as "no checkpoint" (graceful degradation)
    • Checkpoint does NOT duplicate existing extracted facts — each fact is stored once; checkpoint just references their memory IDs
    • bun test: 12 new tests pass; 0 regressions on existing 650
    • bun run typecheck: 0 errors
    • All 4 Go modules build + test clean
    • Dependencies: T-026 (compaction) and T-027 (session manager) — both DONE. No new P2 dependencies.
    • Effort: 1.0 day
    • Parallelization: Can run in parallel with T-083 (zero shared files). Blocks T-080, T-081, T-082, T-084, T-085 (all need checkpoint persistence to exist). Coder-only — no architect needed; the schema is straightforward.

The TUI picks up exactly where it left off.

    Task T-081: Offline Mode (Read-Only / TUI-Only)¶

    • Goal: Make the TUI functional with reduced capabilities when the OpenCode server OR Go backend is unavailable. In offline mode, the TUI provides: TASKS.md browser (read/edit cards locally), memory browser (read from saved checkpoints — no writes), ChainPanel (replays cached chains only), SpendPanel (shows last-known values from checkpoint), and OpportunityPanel (shows cached findings — no new detections). Chat panel shows "🟧 OFFLINE — Backend unavailable. Local operations only." The TUI is NEVER a dead terminal — it degrades gracefully.
    • Scope IN:
    • packages/tui/src/opencode-client/client.ts — enhance degraded mode detection: if status === "offline" or status === "degraded", emit offlineState event with reason ("OpenCode server unreachable", "Go backend crashed", "both"). Add getOfflineState(): OfflineState getter.
    • packages/tui/src/neuralgentics-client/client.ts — enhance offline detection: if backendProcess === null or waitForReady() times out, set status: "offline". Emit event. Add getOfflineState().
    • packages/tui/src/index.ts — add OfflineBanner overlay component: renders "🟧 OFFLINE" banner at top of screen with reason. Wire into main render loop — check offlineState on every render tick, show/hide banner. Block all write operations (prompt send, memory.add, dispatch) when offline. Enable read operations (memory.query, kanban parse, chain replay, spend read, opportunity read).
    • packages/tui/src/panels/status.ts — extend StatusBarData with offlineState: OfflineState. Show "LLM:offline" or "Backend:offline" in status bar.
    • packages/tui/src/commands.ts — add /offline command: shows offline state details (what's down, what's available, last checkpoint age). Gate write commands: /compact, /scaffold (write), /resume (write) → blocked with "Unavailable in offline mode" message.
    • packages/tui/src/sidecar.ts — add checkSidecarHealth(): Promise<boolean> for offline mode diagnostics. Report in /offline.
    • packages/tui/src/__tests__/offline-mode.test.ts — tests for offline detection, banner rendering, command gating, graceful degradation.
    • Scope OUT:
    • Auto-reconnection logic (retry-with-backoff on backend recovery) — P3 hardening
    • Offline memory editing (TASKS.md is editable in offline mode because it's a file; neuralgentics memories are read-only because the backend is the write path)
    • Offline opportunity detection (→ T-085 provides CACHED results; new detection requires the backend/LLM)
    • Offline dispatch or sub-agent spawning (obviously requires OpenCode)
    • Sub-tasks:
    • packages/tui/src/opencode-client/client.ts — enhance OpenCodeClient with offlineState event emitter, getOfflineState() getter, reason propagation (~40 lines add)
    • packages/tui/src/neuralgentics-client/client.ts — enhance NeuralgenticsClient with offlineState, health check retry loop, event emitter (~50 lines add)
    • packages/tui/src/index.ts — add OfflineBanner component (absolute-positioned text overlay), wire into render loop, gate write operations, route reads to cached data (~80 lines add)
    • packages/tui/src/panels/status.ts — add offlineState to StatusBarData, render "LLM:offline"/"Backend:offline" (~15 lines add)
    • packages/tui/src/commands.ts — add /offline command handler: display offline state details, gate write commands (~50 lines add)
    • packages/tui/src/sidecar.ts — add checkSidecarHealth() (~30 lines add)
    • packages/tui/src/__tests__/offline-mode.test.ts — 15 tests (offline detection from OpenCodeClient, offline from NeuralgenticsClient, banner rendering, command gating for /compact, /scaffold, /resume, read operations allowed, /offline command output, sidecar health check, double-offline state, recovery detection, state transitions)
    • Acceptance criteria:
    • Go backend process exits → TUI detects within 5 seconds → status bar shows "Backend:offline"
    • OpenCode server port unreachable → TUI detects within 5 seconds → status bar shows "LLM:offline"
    • Both down → banner "🟧 OFFLINE — Backend + LLM unavailable. Local operations only."
    • Offline TUI can read TASKS.md (kanban panel works), browse saved memories (from checkpoints), and view cached chain/spend/opportunity data

    • bun test: 15 new tests pass; 0 regressions
    • bun run typecheck: 0 errors

    Task T-083: 3-Way Merge Viewer¶

    • Goal: Extend the 2-way diff panel (panels/diff.ts, 710 lines) to support 3-way merge: base (original), ours (local), theirs (coder's proposed changes). Render three side-by-side panes in a split layout. Add keybindings: Tab to cycle active pane, 1/2/3 to jump directly, y/n to accept/reject. This upgrades the T-030 simple diff panel to a proper merge-conflict resolution tool.
    • Scope IN:
    • packages/tui/src/panels/diff.ts — extend DiffPanel with renderThreeWay(base: string, ours: string, theirs: string): void. The 3-way mode uses a horizontal split: left pane (base, read-only, dimmed), center pane (ours, active by default, highlighted), right pane (theirs, selectable, highlighted). State machine: activePane: 'ours' | 'theirs' | 'base', resolved: boolean. On y accept: resolve with ours or theirs depending on active pane. On n reject: discard theirs, keep ours. On Tab: cycle ours → theirs → base → ours. On 1/2/3: jump directly to base/ours/theirs.
    • packages/tui/src/panels/diff.ts — add DiffMode enum (twoWay, threeWay). Extend renderDiffPanel() to accept mode parameter and render accordingly. The 2-way mode is unchanged; 3-way is additive.
    • packages/tui/src/index.ts — wire showDiffPanel to pass a mode parameter from the triggering command (2-way from /diff, 3-way from future merge conflict detection or /diff --threeway).
    • packages/tui/src/commands.ts — extend /diff to support /diff --threeway base ours theirs or /diff --merge (detect merge state automatically). When merge state detected, call renderThreeWay().
    • packages/tui/src/__tests__/three-way-merge.test.ts — tests for 3-way layout, pane navigation, accept/reject with active pane selection, empty base, conflict markers, edge cases.
    • Scope OUT:
    • Automatic merge conflict detection (detecting actual git conflicts in working tree) — P3
    • Inline editing of the merge result (read-only viewer for P2; edit is v0.2.0)
    • 3-way merge for files other than code diffs (only code for now)
    • Integration with git merge (purely a viewer — the git merge happens outside the TUI)
    • Sub-tasks:
    • packages/tui/src/panels/diff.ts — add DiffMode enum, ThreeWayState interface, renderThreeWay() method, extend renderDiffPanel() with mode dispatch, add pane navigation (Tab/½/3 keybindings), add accept/reject logic with active pane awareness (~250 lines add to existing 710L)
    • packages/tui/src/commands.ts — extend /diff parser with --threeway flag + args (~30 lines add)
    • packages/tui/src/index.ts — wire mode parameter into showDiffPanel (~15 lines add)
    • packages/tui/src/__tests__/three-way-merge.test.ts — 12 tests (3-way layout renders, pane navigation Tab cycle, pane jump ½/3, accept ours, accept theirs, reject theirs, base pane is read-only, empty base edge case, large diff rendering, single-line change, conflict markers handling, revert to 2-way mode)
    • Acceptance criteria:
    • renderThreeWay(base, ours, theirs) renders three side-by-side panes with correct content
    • Tab cycles pane highlight: ours → theirs → base → ours
    • 1 jumps to base pane (read-only), 2 to ours, 3 to theirs
    • Active pane on theirs + y → accept theirs (resolve merge with their version)

    • Existing 2-way /diff still works — no regression on T-030's panel
