    Session 29 Container Architecture Design¶

    Neuralgentics v0.2.0 — Docker/Podman Container Stack¶

    Status: Design complete (Architect phase)
    Target: Coder Context Package for T-FOLLOWUP-CONTAINERS-ENABLE
    Date: 2026-06-06
    Branch: feat/containers (to be created by coder)

    1. Overview & Goals¶

    What We're Shipping in v0.2.0¶

    Four working container images, built and pushed to GHCR by CI, pullable via docker-compose up or podman-compose up:

    Image GHCR Tag Purpose
    ghcr.io/veedubin/neuralgentics-postgres v0.2.0 PostgreSQL 18 + pgvector extension
    ghcr.io/veedubin/neuralgentics-sidecar v0.2.0 Python gRPC embedding service
    ghcr.io/veedubin/neuralgentics-backend v0.2.0 Go JSON-RPC backend (distroless)
    ghcr.io/veedubin/neuralgentics-tui v0.2.0 Terminal UI (compiled Bun binary, distroless)

    Both docker-compose up and podman-compose up work end-to-end: - Postgres with schema auto-applied on first start. - Sidecar loads the embedding model on first startup (persisted via named volume). - Backend connects to postgres + sidecar, exposes JSON-RPC over stdio. - TUI connects to backend for interactive use.

    Why Now¶

    The current install path requires 5 dependencies: 1. Go toolchain (for building the backend) 2. Node/Bun (for building the TUI) 3. Podman or Docker (for running the sidecar) 4. SSL certificates (manual setup for TLS) 5. Manual sidecar startup

    podman-compose up reduces this to 1 dependency (Podman or Docker). Users pull images from GHCR and run a single command.

    What This Is NOT¶

    • NOT a production Kubernetes deployment.
    • NOT a multi-node / high-availability setup.
    • NOT a scaling story (single postgres, single sidecar, single backend).
    • NOT an SSL/TLS-enabled deployment (containers communicate on internal bridge network; no encryption needed).

    2. Base Image Decisions¶

    PostgreSQL + pgvector: pgvector/pgvector:pg18¶

    User-locked decision from Session 28. Not revisitable.

    • Rationale: The official pgvector image keeps postgres + pgvector extension in lockstep. pg18 is the current stable major version. Upgrading now avoids tech debt down the road (pg17 EOL).
    • How schema gets baked in: PostgreSQL's Docker entrypoint auto-executes any .sql or .sh files found in /docker-entrypoint-initdb.d/ on first container start, in alphabetical order. We copy all *.up.sql migration files into that directory. The entrypoint runs them against POSTGRES_DB.
    • SHA pin decision: DROP the SHA digest pin. The current postgres.Dockerfile pins @sha256:8a6a4e93... (a pg17 digest). For the rewrite, use pgvector/pgvector:pg18 (tag-only). Official Docker Hub images are rarely retagged maliciously; the operational burden of updating SHA pins on every minor release outweighs the supply-chain benefit for a single-node local install. If the team later needs strict reproducibility (e.g., regulated environments), the SHA can be re-added by commenting # Pin with: podman inspect pgvector/pgvector:pg18 --format='{{.RepoDigests}}'.
    • Existing Dockerfile reference: docker/postgres.Dockerfile (12 lines, current). Lines 2–4 contain the old base + SHA.

    Sidecar: python:3.11-slim-bookworm¶

    Unchanged from current. The existing docker/sidecar.Dockerfile already uses multi-stage with this base. Python 3.11 is the LTS version with the broadest ML library support. slim-bookworm is smaller than full Debian but has the C toolchain needed for grpcio native extensions during the build stage.

    Backend: gcr.io/distroless/static-debian12:nonroot¶

    Unchanged. The Go binary is statically linked (CGO_ENABLED=0), so a shell-less distroless image is ideal: minimal attack surface, no package manager, no shell injection risk. ca-certificates are copied from the builder for TLS connections.

    TUI: gcr.io/distroless/static-debian12:nonroot¶

    Unchanged. bun build --compile produces a standalone binary, so distroless is sufficient.

    3. Multi-stage Build Structure¶

    postgres.Dockerfile (single-stage)¶

    FROM pgvector/pgvector:pg18

COPY packages/memory/src/neuralgentics/memory/store/migrations/postgres/*.up.sql \
     /docker-entrypoint-initdb.d/

ENV POSTGRES_DB=neuralgentics

EXPOSE 5432

HEALTHCHECK --interval=5s --timeout=3s --start-period=10s --retries=10 \
    CMD pg_isready -U postgres -d neuralgentics

    Changes from current (v0.1.0): - Base image: pgvector/pgvector:pg17@sha256:... → pgvector/pgvector:pg18 - Added explicit EXPOSE 5432 - Added inline HEALTHCHECK directive (currently the healthcheck is only in docker-compose.yml) - Dropped SHA digest pin (rationale in Section 2)

    Migration files copied (4 files, executed in order): 1. 000001_initial_schema.up.sql — core memories, peers, knowledge graph tables 2. 000002_project_chunks.up.sql — project indexing tables 3. 000003_memories_1024.up.sql — dual-model 1024-dim embedding table 4. 000004_agent_tools.up.sql — agent tool registration

    sidecar.Dockerfile (2-stage: builder + slim)¶

    Stage 1 — Builder: 

    FROM python:3.11-slim-bookworm AS builder

RUN apt-get update && \
    apt-get install -y --no-install-recommends build-essential && \
    rm -rf /var/lib/apt/lists/*

RUN pip install --no-cache-dir uv

WORKDIR /app

COPY packages/memory/cmd/embedding-sidecar/requirements.txt .
RUN uv pip install --system --no-cache -r requirements.txt

    Stage 2 — Final: 

    FROM python:3.11-slim-bookworm

COPY --from=builder /usr/local/lib/python3.11/site-packages/ /usr/local/lib/python3.11/site-packages/

WORKDIR /app

COPY packages/memory/cmd/embedding-sidecar/main.py .
COPY packages/memory/cmd/embedding-sidecar/embedding_sidecar/ embedding_sidecar/
COPY packages/memory/cmd/embedding-sidecar/proto/ proto/

EXPOSE 50051

ENV PYTHONPATH=/app
ENV NEURAL_EMBED_ADDR=0.0.0.0:50051

HEALTHCHECK --interval=10s --timeout=7s --start-period=30s --retries=5 \
    CMD python -c "import grpc; ch=grpc.insecure_channel('localhost:50051'); grpc.channel_ready_future(ch).result(timeout=5)" || exit 1

CMD ["python", "main.py"]

    Changes from current (v0.1.0): - Dropped SHA digest pins on both FROM lines (tag-only) - No functional changes — the existing structure is already correct

    Result size: The builder includes build-essential (~200 MB) for compiling native extensions (grpcio, sentence-transformers). The final image is ~400-600 MB (most of which is the Sentence-Transformers model downloaded at runtime). The 2-stage split prevents build tools from bloating the final image.

    backend.Dockerfile (2-stage: golang builder + distroless)¶

    Stage 1 — Builder: 

    FROM golang:1.24-bookworm AS builder

WORKDIR /src

COPY go.work go.work.sum ./

COPY packages/memory/ packages/memory/
COPY packages/orchestrator-go/ packages/orchestrator-go/
COPY packages/broker-go/ packages/broker-go/
COPY packages/backend-go/ packages/backend-go/

WORKDIR /src/packages/backend-go
RUN go mod download

ARG VERSION=v0.2.0
RUN CGO_ENABLED=0 go build \
    -ldflags="-s -w -X main.version=${VERSION}" \
    -o /app/neuralgentics-backend ./cmd/backend/

    Stage 2 — Final: 

    FROM gcr.io/distroless/static-debian12:nonroot

COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

COPY --from=builder /app/neuralgentics-backend /usr/local/bin/neuralgentics-backend

EXPOSE 8080

ENTRYPOINT ["neuralgentics-backend"]

    Changes from current (v0.1.0): - Dropped SHA digest pins on both FROM lines - Default VERSION arg bumped to v0.2.0 - No functional changes

    Binary properties: CGO_ENABLED=0 produces a fully static binary. -ldflags="-s -w" strips debug symbols and the DWARF table, reducing binary size ~30%. -X main.version=${VERSION} injects the release tag for version reporting.

    tui.Dockerfile (2-stage: bun builder + distroless)¶

    Stage 1 — Builder: 

    FROM oven/bun:1.3-slim AS builder

WORKDIR /app

COPY packages/tui/package.json packages/tui/bun.lock ./
RUN bun install --frozen-lockfile

COPY packages/tui/src/ src/
COPY packages/tui/tsconfig.json ./

ARG VERSION=v0.2.0
RUN bun build --compile \
    --target=bun-linux-x64 \
    --outfile=dist/neuralgentics \
    src/index.ts

    Stage 2 — Final: 

    FROM gcr.io/distroless/static-debian12:nonroot

COPY --from=builder /app/dist/neuralgentics /usr/local/bin/neuralgentics

ENTRYPOINT ["neuralgentics"]

    Changes from current (v0.1.0): - Dropped SHA digest pins on both FROM lines - Default VERSION arg bumped to v0.2.0 - No functional changes

    4. Networking Decisions¶

    Topology¶

    ┌───────────────────────────────────────┐
│  Host (localhost)                      │
│                                       │
│  Port 6000 ──→ postgres:5432          │
│                 (host-accessible)      │
│                                       │
│  podman-compose run --rm tui           │
│    └─→ stdin/stdout → backend          │
│                           │            │
│  bridge: neuralgentics-net            │
│  ┌───────────┬──────────┬──────────┐  │
│  │ postgres  │ sidecar  │ backend  │  │
│  │ :5432     │ :50051   │ :8080    │  │
│  └───────────┴──────────┴──────────┘  │
└───────────────────────────────────────┘

    Rules¶

    1. Host port 6000 → postgres container port 5432. Per AGENTS.md constraint: "Port 6000 only." This is the sole externally exposed port.
    2. Internal services communicate on the bridge network: neuralgentics-net. Sidecar, backend, and postgres resolve each other by service name (e.g., neuralgentics-postgres:5432).
    3. TUI is interactive (no port exposed). Users run podman-compose run --rm neuralgentics-tui (or docker compose run --rm neuralgentics-tui). The TUI attaches to the backend over stdio (JSON-RPC). It's also commented out by default in the compose file (see Section 15, Open Questions).
    4. Startup ordering via depends_on:
    5. Backend: depends_on: { neuralgentics-postgres: { condition: service_healthy }, neuralgentics-sidecar: { condition: service_healthy } }
    6. TUI: depends_on: [neuralgentics-backend] (list form for max compatibility — see Section 13, Risks)

    docker-compose.yml vs. podman-compose.yml Compatibility¶

    docker-compose v2 supports depends_on with condition: service_healthy. podman-compose v1.x only supports the list form (depends_on: [service-name]). The main docker-compose.yml uses the dict form with conditions because docker is the primary target. The podman-compose.yml (see Section 9) uses the list form.

    5. Volume Strategy¶

    Volume Name Container Path Purpose Lifecycle
    neuralgentics-db /var/lib/postgresql/data PostgreSQL data directory (tables, indexes, WAL) Persists across down/up cycles. Manual podman volume rm neuralgentics_neuralgentics-db to reset.
    sidecar-cache /root/.cache/huggingface Sentence-Transformers model cache (~2 GB for all-MiniLM-L6-v2) Persists across restarts. First startup downloads the model; subsequent starts load from cache.
    (none) (backend) Backend is stateless — all state is in postgres. Container can be destroyed and recreated at any time.
    (none) (tui) TUI is stateless. No persistent state. Container is ephemeral.

    Host paths: None. Both volumes are Docker/Podman named volumes, managed by the container runtime. No bind mounts. This avoids permissions hell (uid/gid mismatches between host and container).

    6. Health Checks¶

    postgres¶

    pg_isready -U postgres -d neuralgentics
    - Interval: 5s, Timeout: 3s, Retries: 10, Start period: 10s - Rationale: PostgreSQL starts quickly (~5s on first run with empty data directory). pg_isready is the canonical "is the server accepting connections?" check.

    sidecar¶

    python -c "import grpc; ch=grpc.insecure_channel('localhost:50051'); grpc.channel_ready_future(ch).result(timeout=5)"
    - Interval: 10s, Timeout: 7s, Retries: 5, Start period: 30s - Rationale: The sidecar takes longer to start because it downloads the Sentence-Transformers model on first launch (model is ~90 MB compressed). The 30s start period accommodates this. Subsequent starts are faster (model cached).

    backend¶

    The backend is a JSON-RPC-over-stdio service. There is no HTTP health endpoint exposed by default. The "health check" for container orchestration is a process liveness check:

    # docker-compose.yml
healthcheck:
  test: ["CMD", "pgrep", "neuralgentics-backend"]  # distroless has no shell

    Problem: gcr.io/distroless/static-debian12:nonroot has no /bin/sh, so CMD-SHELL doesn't work. CMD execs directly. However, pgrep is also not in distroless.

    Decision: For v0.2.0, we use the docker-compose depends_on condition mechanism to gate startup ordering. The backend's health status is derived from:

    1. Backend starts after postgres + sidecar are healthy (via depends_on conditions).
    2. If the backend process exits, the container stops, and docker-compose detects it.

    Future enhancement (v0.2.1+): Add a minimal /healthz HTTP endpoint to the backend that returns 200 when postgres + sidecar gRPC are reachable. Then the healthcheck becomes: 

    HEALTHCHECK --interval=10s --timeout=5s --retries=3 \
    CMD ["curl", "-f", "http://localhost:8080/healthz"]
    But curl isn't in distroless either. Solutions: (a) switch to gcr.io/distroless/static-debian12:nonroot + add a minimal Go health binary, or (b) switch to a :debug distroless variant (has busybox). Decision: defer to v0.2.1.

    tui¶

    No health check. The TUI is an interactive process. If the user wants to quit, they Ctrl+C (or podman-compose down). Restart policy is "no".

    Secrets Management¶

    • .env file: The canonical place for local credentials. .env is already in .gitignore (line 42–45). A compose.example.env file ships in the release with placeholder values.
    • GHCR secrets: secrets.GITHUB_TOKEN (auto-provided by GitHub Actions) is used for image push auth. No additional secrets needed.
    • NEVER hardcode POSTGRES_PASSWORD in docker-compose.yml. The current compose file defaults to neuralgentics which is fine for local development (it's a well-known default, not a security boundary). The example env file should document this.

    Compatibility¶

    • docker compose v2: Fully supported. Uses depends_on with condition: service_healthy.
    • docker-compose v1: May produce warnings on depends_on condition (not supported in v1). Primary target is v2.
    • podman-compose: See Section 9 for the dedicated podman-compose.yml.

    9. podman-compose.yml¶

    Podman has native features that docker doesn't support. Rather than clutter the main compose file with conditional env vars (PODMAN_COMPOSE=1 → enable SELinux labels), we ship a separate file.

    Podman-Specific Features¶

    Feature Config Why
    SELinux labels :Z suffix on volume mounts Fedora/RHEL/CentOS Stream enforce SELinux. Without :Z, containers can't write to named volumes. :Z relabels the volume for container access.
    User namespace userns_mode: "keep-id" Maps the host user's UID into the container, avoiding permission issues on rootless podman.
    Process limits pids_limit: 1000 Prevents runaway fork bombs or goroutine leaks from consuming system PIDs.
    depends_on List form: depends_on: [service-name] podman-compose v1.x doesn't support condition: service_healthy. Startup ordering is best-effort.

    Decision: Separate File¶

    Two files is cleaner than one conditional file. Rationale: - The differences are significant (3 podman-only directives per service, plus different depends_on syntax). - Conditional env vars (PODMAN_COMPOSE=1) would make the file harder to read and test. - The compose.sh script already detects the runtime; it can select the right file automatically.

    compose.sh Update (out of scope for coder, handled by orchestrator)¶

    The scripts/compose.sh wrapper should check for PODMAN_COMPOSE=1 env var or auto-detect podman and use podman-compose.yml instead of docker-compose.yml. This is a small follow-up; the coder can add a TODO comment in the compose script.

    13. Risks & Mitigations¶

    Risk 1: pgvector/pgvector:pg18 May Not Exist¶

    Likelihood: Low. pg18 is the current stable PostgreSQL major version. The pgvector/pgvector image tracks PostgreSQL releases closely.

    Mitigation: Coder verifies the tag exists before rewriting: 

    podman pull pgvector/pgvector:pg18
    If the tag doesn't exist or the pull fails, fall back to pgvector/pgvector:pg17 and document the reason in the design doc's Open Questions section.

    Risk 2: Distroless Image Can't Run the Binary¶

    Likelihood: Low. The Go backend uses CGO_ENABLED=0 (static linking). The TUI uses bun build --compile. Both produce standalone binaries with no dynamic library dependencies (beyond libc, which distroless includes).

    Mitigation: Coder verifies the binary runs in distroless: 

    podman run --rm -it gcr.io/distroless/static-debian12:nonroot /bin/sh
# Expected: exec: "/bin/sh": stat /bin/sh: no such file or directory
# This confirms NO shell is available — as intended.
    The backend binary has already been tested in distroless during v0.1.0 development (it shipped in the first compose file).

    Risk 3: GHCR Auth Fails on First Push¶

    Likelihood: Medium. The secrets.GITHUB_TOKEN has packages: write scope by default, but the job-level permissions block may shadow the root permissions.

    Mitigation: Add explicit permissions: { contents: read, packages: write } at the job level (line 319+). This is the documented pattern for docker/build-push-action.

    Risk 4: podman-compose v1.x Doesn't Support depends_on: condition:¶

    Likelihood: High. podman-compose v1.x supports only the list form depends_on: [service-name].

    Mitigation: The podman-compose.yml uses the list form exclusively. Startup ordering is best-effort; the backend may start before postgres is fully ready. The backend already has connection retry logic (it will exit and restart if postgres isn't available). restart: unless-stopped handles this gracefully.

    Risk 5: Sidecar Model Download Fails on First Run¶

    Likelihood: Low. Hugging Face has high uptime. The Sentence-Transformers library has a local cache fallback.

    Mitigation: The sidecar's start_period is 30s (ample for the model download). If it fails, restart: unless-stopped retries. The sidecar-cache volume persists the model for subsequent starts.

    15. Open Questions for Coder¶

    Q1: SHA Digest Pins — Pin or Trust Tags?¶

    Recommendation: Trust the tags for v0.2.0. Add a comment in each Dockerfile noting how to pin if reproducibility is needed:

    # To pin a specific digest for strict reproducibility:
#   podman inspect pgvector/pgvector:pg18 --format='{{.RepoDigests}}'
FROM pgvector/pgvector:pg18

    Rationale: SHA pins are a maintenance burden — every minor release of postgres/python/golang/bun requires updating the pin. For a single-node local install, the security benefit is negligible. If this were a multi-tenant SaaS deployment, we'd pin. It's not.

    Q2: TUI Service — Optional or Default?¶

    Recommendation: Comment out the TUI service by default in both docker-compose.yml and podman-compose.yml. Add a comment block:

      # ── TUI (commented out by default — uncomment to use interactively) ──
  # To use: uncomment this block and run `podman-compose run --rm neuralgentics-tui`
  # Most users want the backend API, not the TUI.

    Rationale: The TUI is an interactive tool for developers. The podman-compose up command is for production-like environments where the backend runs as a service. The TUI is an "attach and explore" tool, not a service. Commenting it out by default avoids confusion (users seeing 5 services when they expected 4).

    Q3: compose.example.env — Ship as Committed File or Generate?¶

    Recommendation: Ship as a committed file. It's a template with documented placeholders, not a secret.

    # compose.example.env — Copy to .env and customize
NEURALGENTICS_VERSION=v0.2.0
NEURALGENTICS_DB_PORT=6000
NEURALGENTICS_DB_USER=postgres
NEURALGENTICS_DB_PASSWORD=neuralgentics   # Change for production!
NEURALGENTICS_DB_NAME=neuralgentics
POSTGRES_PASSWORD=neuralgentics           # Must match NEURALGENTICS_DB_PASSWORD
NEURALGENTICS_EMBED_DEVICE=cpu
EMBEDDING_MODE=auto
THOUGHT_CHAINS=true

    The file is committed to the repo, shipped in the release assets, and users cp compose.example.env .env && $EDITOR .env.

    Q4: Backend Health Check — Add /healthz Now or Defer?¶

    Recommendation: Defer to v0.2.1. The backend currently has no HTTP endpoint. Adding a /healthz handler requires: 1. A new Go HTTP server in the backend binary (trivial — 20 lines). 2. Adding curl or a custom binary to the distroless image, OR switching to :debug variant (adds busybox, ~5 MB).

    The complexity-to-value ratio is too high for v0.2.0. The depends_on conditions on postgres + sidecar handle startup ordering. If the backend crashes, the container exits.

    Q5: Should We Keep the version: "3.9" Top-Level Key in compose Files?¶

    Recommendation: Remove it. Docker compose v2 (the current standard) ignores it and prints a deprecation warning. podman-compose ignores it silently. It's technical debt with no benefit.
