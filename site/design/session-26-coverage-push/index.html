    Session 26 — Coverage Push: God-File Split + Untested Files + RRF Gap¶

    Date: 2026-06-05 Phase: Coverage Stabilization Theme: De-risk the 5.5%-coverage store/ package by splitting the 1,773-line god-file Total Estimated Effort: ~6 days (11 cards, 0.5–1 day each) Baseline: neuralgentics v0.1.0, commit e74b7f2, working tree clean

    Background¶

    The store/ package is the data backbone of neuralgentics — every memory operation, relationship, entity, peer, thought chain, audit event, project chunk, and security query goes through PostgresStore. Yet the package has 5.5% test coverage (Session 23 audit, docs/design/session-23-coverage-audit.md), and its heart is a 1,773-line god-file (memories.go) with 78 methods spanning 10+ domains. This is the #1 priority from the Session 23 architect review (docs/design/session-23-architect-review.md, §3, Ranked Refactor Backlog #1).

    Why Now¶

    Metric Current Target
    store/memories.go LOC 1,773 ~230 after splits (CRUD section only)
    store/ package coverage 5.5% ≥80%
    God-file domains in one file 10 1 (CRUD naturally remains)
    Untested store/ files with dedicated tests 2 (vector.go + partial memories) 6+ (all source files)

    Safety Property¶

    The PostgresStore struct is defined in postgres.go:24. Go allows methods on a type to be defined in any file in the same package. Every extraction is pure code movement — zero logic changes, zero signature changes, zero import changes (all files stay in package store). The only risk is merge conflicts if two extractions modify overlapping line ranges in memories.go simultaneously. See §Parallelization for the safe strategy.

    Caller Graph (verified grep)¶

    All callers access methods via m.store.MethodName() (through the memory.go facade) or s.store.MethodName() (in search/hybrid.go). No caller imports individual files; all access is through the *PostgresStore receiver. Splitting methods into separate files preserves the receiver and method names — zero caller changes needed.

    Why sequential for T-COV-001 through T-COV-008: Each extraction removes lines from memories.go and adds them to a new file. Removing lines shifts all line numbers above. Extracting bottom-first means each card's line references in memories.go remain stable. Parallel extraction of non-overlapping regions is possible (see §Parallelization) but adds merge-conflict risk.

    T-COV-009, T-COV-010, T-COV-011 are parallelizable — they touch different files, no shared state.

    Parallelization Batches¶

    Batch 1: Bottom + Top (2 parallel extractions, if safe)¶

    Card Region Lines Rationale
    T-COV-001 Bottom (1529–1621) 93 Extracts from end of file; removing these doesn't shift anything above
    T-COV-008 Top (230–335) 106 Extracts from upper-middle; doesn't overlap with bottom region

    Constraint: After Batch 1, memories.go has two "holes" — lines 230–335 and 1529–1621 are removed. Remaining contiguous regions: 1–229 (CRUD), 336–1528 (entities through project chunks), 1622–1773 (profiles). The line numbers for the middle region shift by -106 (relationship extraction above them). This is manageable but requires the Batch 1 coder to document the new line offsets.

    Risk with Batch 1 parallel: If T-COV-001 and T-COV-008 both modify memories.go in parallel, git may not merge cleanly (both remove lines from the same file). Recommendation: Run them sequentially unless both coders coordinate through a shared intermediate commit.

    Batch 2: Sequential Middle Sweep (6 sequential extractions)¶

    After Batch 1, extract remaining domains bottom-to-top: 1. T-COV-002 (profiles — now at ~1622-end of remaining file) 2. T-COV-003 (project chunks) 3. T-COV-004 (audit+trust) 4. T-COV-005 (thoughts — largest, 1 day) 5. T-COV-006 (peers) 6. T-COV-007 (entities)

    Batch 3: Parallel Test Cards (3 parallel)¶

    After all extractions, all 3 test cards can run simultaneously: - T-COV-009 (postgres + agent_tools tests) - T-COV-010 (search.go tests) - T-COV-011 (RRF coverage in hybrid.go)

    Risk Register¶

    # Risk Severity Mitigation
    1 Merge conflicts on memories.go from parallel extractions MEDIUM Use sequential extraction (bottom-to-top) or coordinate through shared intermediate commits. Each card commits its extraction before the next card starts.
    2 Helper functions (scanThoughtRow, derefString, formatVector) lose accessibility LOW All helpers are package-level (unexported, lowercase first letter). They remain accessible from any file in package store. No change needed. If formatVector is needed by both project_chunks.go and memories_1024.go, move it to a shared store/vector_util.go.
    3 InferenceChain BFS traversal breaks on extraction LOW Pure code movement — BFS algorithm doesn't change. However, the existing code was untested before extraction; any pre-existing BFS bug becomes visible when tests are added.
    4 GetTrustAdjustments + ListFadingMemories stub methods LOW These already return fmt.Errorf("not implemented"). The Session 23 fix (replacing nil,nil with error returns) is already applied. Tests should assert the error is returned.
    5 CountMemories broken primary query LOW Session 23 architect smell S5: QueryRow(ctx, GetMemoryCount).Scan(nil, &active, nil) scans into nil columns. The fallback raw SQL works. This is a pre-existing bug, not introduced by extraction. Fix inline during T-COV-008 or add a follow-up card.
    6 Silent error swallowing in 7+ scan loops MEDIUM Session 23 architect smell S2: if err := rows.Scan(...); err != nil { continue } in relationships.go, entities.go, peers.go, project_chunks.go. The extraction cards do NOT change logic, so this smell persists. Covered by T-COV-008 acceptance test TestGetRelationships_MalformedRow — but the fix itself (changing continue to error propagation) should be in a separate bugfix card.
    7 Testcontainers dependency for DB-backed tests MEDIUM Many tests (search.go, postgres.go, thought chain tests) need a pgx pool. The project uses testcontainers-go; ensure CI has Docker available. Alternatively, use mockpgx for lighter tests.
    8 Existing test fixtures (memories_count_test.go, memories_get_test.go) reference PostgresStore methods LOW These test files already import store and call methods directly. After extraction, the methods are still on PostgresStore and still in package store. No fixture changes needed.
