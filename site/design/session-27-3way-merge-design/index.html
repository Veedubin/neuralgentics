    Session 27 — 3-Way Merge Viewer Architecture Design¶

    Status: Pre-implementation design (architect-only, read-only) Author: boomerang-architect Date: 2026-06-06 References: T-083 (P2 roadmap §227-261), T-030 (existing diff.ts, 710 lines)

    1. Pane Layout Geometry¶

    Three panes render side-by-side with borders. Base pane is read-only and visually dimmed; ours/theirs are selectable with highlight border.

    ┌─────────────────────── Three-Way Merge ────────────────────────┐
│ ┌── Base (original) ──┬──── Ours (local) ──┬── Theirs (proposed) ──┐ │
│ │                     │                    │                        │ │
│ │ (dimmed, read-only) │ ←ACTIVE→ (default) │   (selectable)         │ │
│ │                     │                    │                        │ │
│ └─────────────────────┴────────────────────┴────────────────────────┘ │
│ ───────────────────────────────────────────────────────────────────── │
│ [Tab] cycle  [1] base  [2] ours  [3] theirs  [y] accept  [n] reject  │
│ [q/Esc] quit without accepting                                        │
└──────────────────────────────────────────────────────────────────────┘

    Width Algorithm¶

    Terminal width >= 80 chars; target layout >= 60 chars per pane group.

    outer_border    = 2          (│ left + │ right)
inner_separators = 2 × 1     (│ between panes)  = 2
total_border_overhead         = 4
content_width = terminal.width - total_border_overhead

base_w   = max(15, floor(content_width × 0.25))  // 25% — smallest, dimmed
ours_w   = floor((content_width - base_w) / 2)   // split remaining evenly
theirs_w = content_width - base_w - ours_w        // ensure all pixels used
    Terminal total_border content base_w ours_w theirs_w
    80 cols 4 76 19 28 29
    100 cols 4 96 24 36 36
    120 cols 4 116 29 43 44

    Minimum terminal: 60 columns. Below 60, fall back to vertical stack (not in scope — raises an error with message "Terminal too narrow for 3-way view (min 60 cols)").

    Pane Visual States¶

    Pane Default border Active border Text rendering
    base dim gray dim gray (no change) dimmed (fg: #888)
    ours white/cyan cyan bold (active highlight) normal
    theirs white/cyan cyan bold (active highlight) normal

    Base pane ignores selection — it is always read-only display. Line numbers gutter: " 123 " (5 chars) on each pane, subtract from content width.

    2. State Machine Transitions¶

    2.1 DiffMode Enum (additive to existing code)¶

    export type DiffMode = "twoWay" | "threeWay";

    DiffPanel stores _mode: DiffMode = "twoWay". The existing show() sets "twoWay"; new showThreeWay() sets "threeWay".

    2.2 ThreeWayState (new, additive)¶

    export type ThreeWayState =
  | "idle"           // panel hidden / not shown
  | "ours-active"    // ours pane has cursor; y=accept ours
  | "theirs-active"  // theirs pane has cursor; y=accept theirs
  | "base-active"    // base pane selected (read-only; y is no-op)
  | "accepted"       // merged with active pane's version
  | "rejected"       // discarded theirs, kept ours
  | "error";         // rendering failure or invalid input

    2.3 Transition Diagram¶

                     showThreeWay()
                      │
          ┌───────────▼────────────┐
          │      ours-active       │◄────────── [default on show]
          │  (ours highlighted)    │
          └──┬──────────┬────┬─────┘
             │ Tab      │ 1  │ 2
    ┌────────▼───────┐  │    │
    │  theirs-active │◄─┘    │
    │ (theirs highlit)│      │
    └──┬────┬────┬───┘      │
       │Tab │ 1  │ 3        │
  ┌────▼────┴──┐ │           │
  │ base-active│◄┘───────────┘
  │(base dimmed,│
  │ read-only)  │
  └──┬────┬─────┘
     │Tab │ 2/3
     │    └──► ours-active / theirs-active
     │
     ├── y ─────► accepted   (ours-active → accept ours; theirs-active → accept theirs)
     │                ⚠ base-active + y → no-op (beep or status: "Base is read-only")
     │
     └── n ─────► rejected   (any active state → "discard theirs, keep ours")
                    q/Esc ──► idle        (any active state → hide panel)

     idle         → ours-active  (showThreeWay called again)

     accepted     → idle         (q/Esc → hide)
     rejected     → idle         (q/Esc → hide)
     error        → idle         (q/Esc → hide)

    Critical invariant: ours-active is the ONLY state where ours is the active pane. theirs-active is the ONLY state where theirs is the active pane. base-active is informational — y does nothing, n still rejects.

    2.4 Existing 2-Way State Machine (UNTOUCHED)¶

    hidden → showing → accepted / rejected / blocked

    The _mode = "twoWay" path uses the ORIGINAL DiffPanelState machine with zero changes. The 3-way state machine runs in a SEPARATE _threeWayState field and only when _mode === "threeWay". handleKey() dispatches on _mode at the top level:

    async handleKey(keyName: string): Promise<boolean> {
  if (this._mode === "threeWay") return this._handleThreeWayKey(keyName);
  return this._handleTwoWayKey(keyName); // existing code, renamed
}

    3. Keybinding Routing¶

    3.1 Key Map¶

    Key ours-active theirs-active base-active accepted/rejected/error
    Tab → theirs-active → base-active → ours-active consumed (no-op)
    1 → base-active → base-active stays base-active consumed (no-op)
    2 stays ours-active → ours-active → ours-active consumed (no-op)
    3 → theirs-active stays theirs-active → theirs-active consumed (no-op)
    y → accepted (ours) → accepted (theirs) no-op + status: "read-only" consumed (no-op)
    n → rejected → rejected → rejected consumed (no-op)
    q → idle (hide) → idle (hide) → idle (hide) → idle (hide)
    Esc → idle (hide) → idle (hide) → idle (hide) → idle (hide)

    All unrecognized keys: consumed (prevent leaking to chat input). Panel is modal.

    3.2 2-Way Key Map (EXISTING, UNCHANGED)¶

    y → accept    n → reject    q/Esc → hide

    No Tab or numeric keys in 2-way mode — these are no-ops (consumed by panel but ignored).

    4. Mode Dispatch Architecture¶

    4.1 Entry Points (additive, no rewrite)¶

                        ┌──────────────────────┐
                    │      DiffPanel        │
                    │  _mode: DiffMode      │
                    │  _threeWayState       │
                    │  _activePaneIdx       │
                    │  _threeWayInput       │
                    ├──────────────────────┤
  twoWay path ───►  │  show()         ◄── existing entry point (T-030)
                    │  handleKey()          │
                    │  _handleTwoWayKey()   │ ← renamed from _handleKey logic
                    ├──────────────────────┤
  threeWay path ──► │  showThreeWay()  ◄── NEW entry point (T-083)
                    │  _handleThreeWayKey() │ ← new key handler with pane routing
                    ├──────────────────────┤
  shared ─────────► │  renderDiffPanel()   │ ← extended with mode param
                    │  _updateDisplay()     │ ← extended for 3-way layout
                    │  createRenderable()   │ ← shared box (visible toggle)
                    └──────────────────────┘

    4.2 renderDiffPanel Signature (extended)¶

    export function renderDiffPanel(
  input: DiffInput,
  title?: string,
  options?: { mode?: DiffMode; activePaneIdx?: 0 | 1 | 2 },
): DiffPanelRenderResult;
    • When mode === "threeWay", calls renderThreeWayLayout(input, options) internally
    • When mode === "twoWay" or absent, runs EXISTING code path (unchanged)
    • Returns same DiffPanelRenderResult shape (lines, additions, removals, state, statusMessage)

    4.3 What the Coder MUST NOT Touch¶

    Do NOT modify Because
    DiffPanelState type 2-way state machine must remain untouched
    show() method body Rename internals to _handleTwoWayKey() but keep logic identical
    Confidence / TestResult types These are 2-way concepts; 3-way has no test callback
    DiffInput interface 3-way uses a SEPARATE ThreeWayDiffInput, not an extension
    createRenderable() box/Text IDs Reuse "diff-panel", "diff-content", "diff-status" IDs
    getWrapUpEvidence() Return null in 3-way mode (no wrap-up evidence for merge viewer)

    5. Data Model¶

    /** Input for the 3-way merge viewer. */
export interface ThreeWayDiffInput {
  /** Original file content before any changes (ancestor). */
  base: string;
  /** Local/current file content (our version). */
  ours: string;
  /** Proposed/coder's file content (their version). */
  theirs: string;
  /** Optional conflict marker lines from a git merge (e.g., "<<<<<<< ours"). */
  conflictMarkers?: string[];
}

/** Stores a single pane's rendered output. */
interface ThreeWayPaneData {
  label: "base" | "ours" | "theirs";
  lines: string[];
  additions: number;
  removals: number;
  isReadOnly: boolean;
  isActive: boolean;
}

    The DiffPanel stores _threeWayPanes: ThreeWayPaneData[] internally. Panes are computed on showThreeWay() by running generateDiffFromBeforeAfter() three times: - Pane 0: base (diff against empty) — renders full file content - Pane 1: ours (diff base→ours) — renders what changed locally - Pane 2: theirs (diff base→theirs) — renders what the coder proposes

    Alternatively (simpler and preferred for v0.2.0): - Each pane renders the FULL file content directly (not as diffs), one line per pane row - Align lines across panes by line number (scroll synced) - This simplifies rendering and avoids 3-way diff computation complexity

    Decision: pure text rendering per pane — each pane renders content.split("\n") directly with line numbers. No diff computation between panes in v0.2.0. Conflict markers (<<<<<<<, =======, >>>>>>>) are displayed as-is if present.

    6. Acceptance Criteria (for coder implementation, T-083)¶

    1. showThreeWay() renders 3 panes — calling panel.showThreeWay({ base, ours, theirs }) with 80-col terminal produces 3 visible panes with base (left, dimmed), ours (center, highlighted), theirs (right, normal) and a keybinding bar at bottom.

    2. Tab cycles correctly — Tab press in any order produces: ours-active → theirs-active → base-active → ours-active. The highlight border moves to the correct pane on each press.

    3. Numeric jumps (½/3) direct — Pressing 1 activates base (read-only, dimmed), 2 activates ours, 3 activates theirs, regardless of current active pane. Edge case: pressing the same number twice keeps the pane active (no flicker).

    4. Accept with active pane — y in ours-active state accepts ours (panel state → accepted, status: "✓ Merged with ours"). y in theirs-active accepts theirs (status: "✓ Merged with theirs"). y in base-active does NOTHING (status: "Base pane is read-only — select ours or theirs to accept").

    5. Reject discards theirs — n from any active state transitions to rejected (status: "✗ Merge rejected — keeping ours"). This is the same behavior regardless of which pane is active.

    6. 2-way mode zero regression — Call panel.show({ diff: "..." }) (or any existing T-030 path) with _mode = "twoWay". y/n/q/Esc produce the EXACT same behavior as before this change. 2-way handleKey returns identical state transitions. All existing diff tests pass without modification.

    7. Narrow terminal graceful degradation — Terminal width < 60 displays an error box: "Terminal too narrow for 3-way view (min 60 cols)" instead of a broken layout. The panel does NOT render garbled or overlapping text.

    8. All keys consumed when panel active — When 3-way panel is visible, ALL key presses (including unrecognized ones) are consumed by handleKey and do NOT leak to the underlying chat input. Modal integrity maintained.

    9. bun test + bun run typecheck — 12 new 3-way tests pass, 0 2-way regressions, 0 type errors.

    7. Risk Mitigation — What the Coder MUST NOT Do¶

    1. DO NOT modify DiffPanelState — the 5-state 2-way machine (hidden | showing | accepted | rejected | blocked) is sacrosanct. Adding a new state like threeWayActive to the existing union type would break the 2-way handleKey switch-case. Create a SEPARATE ThreeWayState type and a _threeWayState field.

    2. DO NOT change the show() method signature — existing callers (/diff command, kanban wrap-up) pass DiffInput. Keep the old signature and add showThreeWay() as a separate method. Do not add mode to show() — use the _mode field as internal dispatch.

    3. DO NOT touch DiffInput interface — 3-way has different data shape (3 strings vs. 1 diff). Create ThreeWayDiffInput. Do not cram base/ours/theirs fields into DiffInput.

    4. DO NOT render 3 panes as 3 separate diff runs — computing pairwise diffs (base→ours, base→theirs, ours→theirs) would balloon complexity. Render each pane as raw file content with line numbers. Pure text, no diff engine.

    5. DO NOT add TestResult or confidence gates to 3-way — the merge viewer is a visual comparison tool, not a verification pipeline. _acceptCallback and confidence logic (low → blocked) are 2-way concepts. 3-way y transitions directly to accepted/rejected without running tests.

    6. DO NOT inline the 3-way key handler into the existing handleKey switch-case — create _handleThreeWayKey() as a separate method. The existing handler should be renamed _handleTwoWayKey() and called via a top-level mode dispatch. This keeps the two state machines visually separate in the source.

    End of design. Handoff to coder for T-083 implementation.
