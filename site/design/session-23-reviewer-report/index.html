    1. Top 10 Critical/High Findings¶

    #1 (CRITICAL) — Broker.DeregisterServer kills shared proxy for ALL servers¶

    • File: packages/broker-go/src/neuralgentics/broker/broker.go:206-216
    • Problem: DeregisterServer calls b.proxy.Stop() which closes the single shared MCPProxy's reader channel and all pending response channels. Since NewBroker() creates exactly one MCPProxy (line 54), stopping it kills the async reader for every server, not just the one being deregistered. ReloadServer (line 146) and ReloadServerWithConfig (line 188) have the same issue.
    • Fix: Per-server proxy tracking. Each StartServer should create a dedicated reader goroutine; DeregisterServer stops only that server's reader. Alternatively, use a reference-counted reader that only stops when the last server deregisters.
    • Test: Add TestBroker_DeregisterServer_DoesNotStopOtherProxies — register two servers, start both, deregister one, verify the other can still receive calls.

    #2 (CRITICAL) — Broker reload test hangs indefinitely (30s timeout)¶

    • File: packages/broker-go/src/neuralgentics/broker/reload_test.go:102
    • Problem: TestReloadServer_RunningServer hangs for 30 seconds and times out. Stack trace shows goroutine stuck in proxy.go:104 (readLoop scanner). The test starts a real subprocess (e.g., cat or sleep), and when the process exits, the scanner blocks on stdout.Read() because the pipe is not closed. The Stop() method closes readerCh but the scanner is blocked on Read(), not on the channel select.
    • Fix: The readLoop at proxy.go:100 uses a non-blocking select on readerCh (line 106-110) but the scanner.Scan() at line 104 blocks the goroutine. When the subprocess exits, scanner.Scan() returns false (EOF), but if the pipe is never closed (e.g., subprocess hangs), the scanner blocks forever. Fix: use io.Copy with a context or set a read deadline on the pipe. Alternatively, close the stdout pipe explicitly in Stop().
    • Test: Already exists — just needs the fix to pass.

    #3 (HIGH) — GetMemory passes includeArchived bool as SQL parameter incorrectly¶

    • File: packages/memory/src/neuralgentics/memory/store/memories.go:94-99
    • Problem: When includeArchived=false, the code passes includeArchived (a bool) as the second query parameter to GetMemoryByID. The query at queries.go:67 is WHERE id = $1 AND ($2::boolean OR is_archived = FALSE). When $2=false, this becomes WHERE id = $1 AND (false OR is_archived = FALSE) which is correct. However, when includeArchived=true, the code uses GetMemoryByIDIncludeArchived (no $2). This is actually correct — the code path is correct for both cases. RETRACTED — the architect flagged this but it's not a bug. The code correctly branches on includeArchived.
    • Verdict: FALSE ALARM — code is correct.

    #4 (HIGH) — Silent error swallowing in 7+ Scan loops¶

    • File: packages/memory/src/neuralgentics/memory/store/memories.go:272,314,381,404,439,1452,1479
    • Problem: Row scanning errors use continue instead of logging or returning. Malformed database rows are silently dropped, making debugging impossible. Affected methods: GetRelationships, GetSupersessionChain, GetEntitiesByType, SearchEntities, GetEntityRelationships, SearchChunks, GetFileChunksByPath.
    • Fix: Replace continue with at minimum slog.Warn("scan error", "err", err) before continue. For critical paths, return the error.
    • Test: Add TestGetRelationships_MalformedRow — insert a row with NULL in a non-nullable column, verify the error is logged (not silently dropped).

    #5 (HIGH) — store package has 5.5% test coverage¶

    • File: packages/memory/src/neuralgentics/memory/store/ (memories.go, postgres.go, queries.go, search.go, agent_tools.go)
    • Problem: The store package is the largest and most critical package (1,768-line god-file + 378-line postgres.go + 653-line queries.go) but has only 5.5% statement coverage. The search package is similarly low at 26.8%. These packages contain all database CRUD operations, relationships, entities, peers, thought chains, audit, trust, project chunks, and 1024-dim operations.
    • Fix: Add unit tests with mock pgx.Rows and pgx.Row interfaces. At minimum, test AddMemory, GetMemory, CountMemories, CreateRelationship, UpsertEntity, AddPeer, StartThoughtChain, LogAuditEvent, AddProjectChunk, SearchChunks.
    • Test: See PART B — Coverage Audit for specific test recommendations.

    #6 (HIGH) — context_builder.go has zero tests¶

    • File: packages/orchestrator-go/src/neuralgentics/orchestrator/context_builder.go (495 lines)
    • Problem: BuildContextPackage, StoreContextPackage, FetchContextPackage, StoreAgentWrapUp, FetchAgentWrapUp — 5 exported functions, 0 tests. This is the core orchestrator logic for building agent context packages.
    • Fix: Add context_builder_test.go with mock MemoryProvider. Test BuildContextPackage with various task types, test StoreContextPackage round-trip, test error paths (memory query failure, marshal failure).
    • Test: TestBuildContextPackage_WithRelevantMemories, TestStoreContextPackage_RoundTrip, TestBuildContextPackage_EmptyMemories.

    #7 (HIGH) — Broker Call has no timeout¶

    • File: packages/broker-go/src/neuralgentics/broker/broker.go:299-330
    • Problem: Call method has no context or timeout parameter. A hung MCP server blocks the broker indefinitely. The underlying proxy.sendRPC has a 30-second timeout (proxy.go:339), but the broker's Call doesn't expose this to callers or allow cancellation.
    • Fix: Add ctx context.Context parameter to Call. Use select with ctx.Done() in sendRPC. This is a breaking API change but necessary for production reliability.
    • Test: TestBroker_Call_ContextCancellation — cancel context before call completes, verify error.

    #8 (MEDIUM) — CountMemories broken primary query¶

    • File: packages/memory/src/neuralgentics/memory/store/memories.go:146
    • Problem: QueryRow(ctx, GetMemoryCount).Scan(nil, &active, nil) — scans into nil for two of three columns. The primary query GetMemoryCount returns 3 columns but only 1 is scanned. The fallback raw SQL at line 149 works correctly.
    • Fix: Fix GetMemoryCount SQL to return only 1 column, or fix the Scan to match the query columns.
    • Test: TestCountMemories_PrimaryQuery — verify the primary query path works (not just the fallback).

    #9 (MEDIUM) — Stub methods return nil,nil (indistinguishable from "no results")¶

    • File: packages/memory/src/neuralgentics/memory/store/memories.go:1302-1304 (GetTrustAdjustments), 1320-1322 (ListFadingMemories)
    • Problem: Both methods return nil, nil with // TODO comments. Callers cannot distinguish "no results" from "not implemented". The ListFadingMemories signature includes threshold float64 and limit int parameters that are completely ignored.
    • Fix: Either implement the queries or return fmt.Errorf("not implemented").
    • Test: TestGetTrustAdjustments_NotImplemented — verify error is returned, not nil.

    #10 (MEDIUM) — Security summary uses string-cast hours parameter¶

    • File: packages/memory/src/neuralgentics/memory/store/memories.go:1705
    • Problem: hoursStr := fmt.Sprintf("%d", hours) converts the integer to a string and passes it as a SQL argument. The queries use $1::integer cast, so PostgreSQL will cast it back, but this is fragile and non-idiomatic. Should pass hours directly as int.
    • Fix: Pass hours as int directly. The SQL queries use $1::integer which accepts both int and string parameters, but passing a string is misleading.
    • Test: TestGetSecuritySummary_HoursParameter — verify the query executes with integer parameter.

    3. Bug Verification¶

    Bug #1: TestProjectIndexer_Index_ConcurrentIndexing — CONFIRMED¶

    • File: packages/memory/src/neuralgentics/memory/index/indexer_test.go:537
    • Status: FAILS 3/3 runs
    • Root cause: The test starts indexing in a goroutine, sleeps 50ms, then tries a second concurrent call. The mock store is too fast (simple map insert with mutex) — the first indexing completes before the 50ms sleep. The atomic.Bool in indexer.go:22-23 works correctly; the test timing is wrong.
    • Fix: Add a synchronization point in the mock (e.g., a channel that blocks AddProjectChunk until the test signals readiness).
    • Adjacent issues: The mock store at indexer_test.go:18-24 has 20+ stub methods for interface compliance. These stubs (lines 622-646) implement GetUserProfile, UpsertUserProfile, GetSecuritySummary, RecordToolRequest, IncrementToolUse, GetAgentTools — all returning nil/nil. This is the same pattern as the production stubs.

    Bug #2: TUI test failures — CONFIRMED (11 failures)¶

    • File: packages/tui/src/__tests__/neuralgentics-client.test.ts
    • Status: 11 tests fail consistently
    • Root cause: The NeuralgenticsClient tests spawn a subprocess and communicate via JSON-RPC over stdio. The tests fail because the subprocess (neuralgentics-backend) either doesn't exist in the test PATH or doesn't produce the expected JSON output. The resolveBackendPath tests fail because the binary path resolution depends on environment variables that may not be set in CI.
    • Fix: Use a mock subprocess (e.g., a script that echoes valid JSON-RPC responses) instead of the real binary. Or set NEURALGENTICS_BACKEND_PATH in the test environment.
    • Adjacent issues: setup.test.ts has 5 tests that all pass. The resolveBackendPath tests in neuralgentics-client.test.ts are the ones failing (2 of the 11).

    Bug #3: Broker.DeregisterServer stops all proxies — CONFIRMED¶

    • File: packages/broker-go/src/neuralgentics/broker/broker.go:206-216
    • Status: Verified by reading proxy.go:Stop() at line 167-182
    • Root cause: NewBroker() at line 54 creates a single MCPProxy instance. DeregisterServer at line 210 calls b.proxy.Stop() which closes p.readerCh (signals reader goroutine to exit) and closes all pending channels. After this, no RPC calls can be made through the proxy. ReloadServer (line 146) and ReloadServerWithConfig (line 188) have the same issue.
    • Fix: Per-server reader tracking. Each StartServer should create a dedicated reader goroutine scoped to that server's stdout pipe. DeregisterServer stops only that server's reader.
    • Adjacent issues: The TestReloadServer_RunningServer test at reload_test.go:102 hangs for 30 seconds and times out. Stack trace shows goroutine stuck in proxy.go:104 (readLoop scanner). This is a separate but related issue — the scanner blocks on stdout.Read() even after Stop() is called because the non-blocking select on readerCh only works between scan iterations.
