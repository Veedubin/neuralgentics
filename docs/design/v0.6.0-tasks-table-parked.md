## Resolved decisions (from the conversation)

1. **Postgres is the durable store** for kanban tasks and parked ideas. Markdown files (TASKS.md, deferred-followups.md) are not the source of truth.
2. **Parking lot is an 8th status** in the same `tasks` table (not a separate table). Parked items still carry context (description, summary, tags, refs) so promotion to a real card is cheap.
3. **Priority is a separate column** from status. Allowed values: `high`, `medium`, `low`, `nice_to_have`.
4. **Auto-summary at insert**: when an idea is added, the model generates a 1-2 sentence summary. Stored alongside rich description. Same pattern as memini-ai tier-1.
5. **TopK at session start**: orchestrator's startup protocol runs a SQL query (or memoryManager tool call) to surface top N high-priority items. Pre-seeded with summaries. Deep query on demand.
6. **Lives in memoryManager** as new MCP tools (not extended onto existing `query_memories`).
7. **TASKS.md fate**: stays for now. User will tell me when to retire it. Don't touch it in this card.
8. **F-1..F-4** (from `docs/design/deferred-followups.md`): migrate into the new table on day one. Delete the .md after migration.

## Open questions (unresolved at park time)

These need user input before implementation. Do NOT proceed without answers.

### Q1. Scope of v0.6.0 vs v0.6.1

Three options were presented. User did not pick one. Options:
- **A. Schema + parking lot only** (v0.6.0). kanban-board-manager skill stays unchanged. ~5 MCP tools, ~1 migration script, ~3 lines in AGENTS.md. Smallest scope, lowest risk.
- **B. Schema + kanban write-through** (v0.6.0). kanban-board-manager skill updated to write to postgres on every state transition. TASKS.md becomes the human-readable view, generated. Larger scope, touches the active kanban.
- **C. Split**: parking lot in v0.6.0, kanban write-through in v0.6.1. Two releases, clean separation.

Default if no answer: A. Reasoning: ships immediate value (parking lot, F-1..F-4 migration) without risk to in-flight work.

### Q2. New `tasks` table — what is the kanban's data model?

The kanban-board-manager skill uses a Markdown card schema with fields: Status, Assignee, Phase, Roadmap, Goal, Acceptance, Scope IN, Scope OUT, Depends on, Blocks, Created, Updated, Wrap-up evidence (changed_files, verification, residual_risk, summary, retry_notes). The current 7 statuses are: triage, todo, ready, running, blocked, done, archived.

The postgres `tasks` table should hold all of these. Need to confirm the field list before writing the migration. Specifically:
- Should `description` be markdown-allowed? (Yes, almost certainly.)
- Should `wrap_up_evidence` be a JSONB column or split into columns? (Suggest JSONB for flexibility.)
- Should `depends_on` / `blocks` be FK columns or just `TEXT[]`? (Suggest TEXT[] of task IDs — simpler, no FK enforcement needed for the parking-lot use case.)

### Q3. Auto-summary generation

The user said "seed a summary that could be generated when it was inserted into the DB by the model when its storing the ideas." This implies the memoryManager tool that does the insert should also call a model to generate a summary. Questions:
- Which model? The same one the user has configured for `small_model`? Or a fixed model? Or the model that called the tool?
- Cost / latency budget? A model call on every insert is non-trivial. Should we make summary generation optional, or have a fast-path (e.g. first 200 chars of description) for cheap mode?
- Can summaries be regenerated later? Probably yes — there should be a `task_regenerate_summary` tool.

### Q4. Session-startup hook

User said: "we probably could just have it part of the new session routine, it just runs an SQL query and gives the agents the data from the this new parking lot. We could do TopK and have it pull the high priority work."

Concretely, this means modifying the orchestrator's startup protocol (in AGENTS.md or the orchestrator persona file) to include a step that queries the parking lot. Need to confirm:
- What's the default TopK? (Suggest 5 H/M/L parked items + 5 ready items.)
- Should this be in AGENTS.md (text instruction to the agent) or in a startup script (actual code that runs)? AGENTS.md is cheaper to change but less reliable.
- Should the result be cached in the agent's context window, or re-queried on demand? Cached is faster but stale; re-queried is fresh but slower.

### Q5. Migration of F-1..F-4

What priority should each F-N get? The .md file says:

- F-2 (unify small_model): "touches opencode core config semantics" — suggests `medium` or `low`
- F-3 (HTTP catalog entries): "needs time to see which hosted MCPs stabilize" — suggests `nice_to_have`
- F-4 (profile conflict UI): "current behavior is the safe default" — suggests `nice_to_have`

Default if no answer: F-1=`low`, F-2=`medium`, F-3=`nice_to_have`, F-4=`nice_to_have`. Reasoning: F-2 is the closest to shipping (it's a pure refactor with no design decisions left); F-1 has the most design ambiguity.
